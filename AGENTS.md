# AGENTS

- PostgreSQL is the primary runtime system of record for operational data.
- Prefer immutable, reproducible runtime artifacts over mutable bind-mounted runtime behavior.
- Keep a single canonical schema definition and governed migration path; conflicting DDL owners are not acceptable.
- For EPL-owned contract boundaries, keep canonical contract schema assets in `tests/contracts/schemas/`; define governed PostgreSQL row compatibility there in one place, and preserve storage compatibility unless an explicit coordination plan says otherwise.
- For MLB-owned contract boundaries, follow the same rule: keep canonical MLB contract schema assets in `tests/contracts/schemas/` (`mlb_*_v1.json`), with deterministic fixture builders in `tests/contracts/fixtures/mlb_*_samples.py` and consumer/provider tests in `tests/contracts/test_mlb_*_{consumer,provider}.py`. MLB-only compatibility fixes belong in MLB-scoped plugin modules (e.g., `plugins/odds_comparator.py` MLB branches, `plugins/bet_loader.py` MLB id/ticker helpers, `plugins/elo/mlb_*.py`, `plugins/mlb_*`); do not expand the MLB contract layer into other sports.
- For Tennis-owned contract boundaries, follow the same rule: keep canonical Tennis contract schema assets in `tests/contracts/schemas/` (`tennis_*_v1.json`), with deterministic fixture builders in `tests/contracts/fixtures/tennis_*_samples.py` and consumer/provider tests in `tests/contracts/test_tennis_*_{consumer,provider}.py`. Tennis-only compatibility fixes belong in Tennis-scoped plugin modules (e.g., `plugins/elo/tennis_elo_rating.py` `predict_with_payload` + Platt calibration, `plugins/odds_comparator.py` TENNIS branches, `plugins/bet_loader.py` TENNIS id/ticker helpers, `plugins/tennis_*`, `plugins/kalshi_markets.py` tennis save path); do not expand the Tennis contract layer into other sports.
